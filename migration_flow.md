# Migration Flow: от статических HTML к клиентским отчётам

Документ описывает целевую архитектуру проекта и план миграции от текущего состояния (статически собираемые HTML по сайтам) к системе автоматической генерации персональных отчётов для клиник.

---

## 1. Задача

Сейчас у нас на ~100 клиник собраны технические аудиты в виде сырых JSON и общий внутренний дашборд. Цель — система, в которой:

- по каждой клинике автоматически собирается **персональный отчёт** на базе единого шаблона;
- остаётся **внутренний дашборд** со всей сводной информацией для нашей команды;
- клиенту отправляется **публичная ссылка** вида `reactortech.ru/<clinic-id>-<token>` + при необходимости PDF-версия;
- при изменении данных по клинике её клиентский отчёт обновляется **автоматически** (без ручной перегенерации файлов);
- дополнять/править шаблон отчёта нужно в одном месте — изменения раскатываются на все 100 отчётов сразу.

Авторизация не требуется — приватность обеспечивается неугадываемым токеном в URL.

---

## 2. Текущее состояние

| Что есть | Где лежит | Роль |
| --- | --- | --- |
| Реестр клиник | `data/sites_manifest.json` | 1 запись на клинику |
| Сырые данные аудита | `data/audits/<id>.audit.json` | HTTP-ответы, заголовки и т.п. |
| Сборщик аудита | `scripts/audit_site.py` | Краулинг и сбор сырья |
| Сборщик HTML | `scripts/build_dashboard.py` | JSON → статические HTML |
| Внутренний дашборд | `dashboard.html` | Сводка по всем клиникам |
| Детальные страницы | `sites/*.html` | По одной на клинику |
| Шаблон отчёта (заготовка) | `final-report-blocks.html` | Статичный HTML, без подстановки данных |
| Локальный сервер | `scripts/dashboard_server.py` | Раздача HTML и сохранение комментариев |

**Главная проблема архитектуры:** сырые `audit.json` напрямую превращаются в HTML. Нет промежуточного нормализованного слоя «находок», поэтому шаблон отчёта нельзя переиспользовать на 100 клиник без боли.

---

## 3. Целевая архитектура

```
audit.json (сырьё)
        │
        ▼
   findings (БД, нормализованный слой)   ◄── единый источник правды
        │
        ├─► внутренний дашборд (/admin)
        │
        └─► клиентский отчёт (/<clinic-id>-<token>)
                       │
                       └─► PDF (через print CSS)
```

### 3.1 Стек

- **Next.js (App Router)** — фронт + бек в одном репозитории.
- **shadcn/ui** — UI-компоненты для шаблона отчёта и дашборда.
- **Prisma + SQLite** на старте; миграция на Postgres при росте нагрузки.
- **Vercel** — хостинг.

Один репо, один деплой. Отдельный бекенд не нужен.

### 3.2 Схема БД

```
clinics
─────────
id (slug, PK)         — например, "dr-knyazkin"
name                  — "Стоматология Князькина"
site                  — "dr-knyazkin.ru"
contact_email
public_token          — случайный токен для публичной ссылки
audit_date
created_at, updated_at

rules                 — справочник видов уязвимостей (одинаков для всех клиник)
─────────
id (slug, PK)         — "no-https", "weak-csp", ...
title                 — заголовок для отчёта
severity              — high / medium / low
description           — объяснение для клиента
recommendation        — что делать

findings              — факт: у клиники X найдена уязвимость Y
─────────
id (PK)
clinic_id (FK → clinics.id)
rule_id   (FK → rules.id)
status                — open / fixed / wont_fix / false_positive
severity              — переопределение severity для конкретного случая (опционально)
evidence              — текст / json: что именно нашли (URL, заголовок, фрагмент кода)
detected_at
updated_at
```

**Правило:** общее описание уязвимости живёт в `rules`, конкретный случай — в `findings`. Не дублировать описание в каждой строке `findings`.

### 3.3 Роутинг

| Путь | Кто видит | Что показывает |
| --- | --- | --- |
| `/` | внутренний | список всех клиник, фильтры, статусы |
| `/admin/<clinic-id>` | внутренний | детальный технический вид + сырьё |
| `/<clinic-id>-<token>` | клиент | публичный отчёт по шаблону |
| `/<clinic-id>-<token>/print` | клиент | print-версия для PDF (Cmd+P) |

### 3.4 Приватность публичных ссылок

URL вида `/<clinic-id>-<token>`, где `token` — случайные 8–12 символов. Угадать невозможно, авторизации не требуется. Это стандартная практика для приватных ссылок (Google Docs «по ссылке», Notion publish).

**Важно:** отчёт содержит описание открытых уязвимостей клиники — фактически инструкцию, как её сломать. Без неугадываемого токена это публиковать нельзя.

### 3.5 PDF

Серверной генерации не делаем. Достаточно качественного `@media print` CSS на странице отчёта — пользователь печатает в PDF через браузер (Cmd+P). При необходимости позже добавим автоматизацию через Playwright, но на старте это лишняя сложность.

## 4. План миграции

### Этап 1. Нормализация данных
1. Спроектировать и заполнить справочник `rules` (список всех видов уязвимостей, которые мы детектируем, с описаниями для клиента).
2. Написать `scripts/extract_findings.py`: читает `data/audits/*.audit.json` → выдаёт нормализованный `findings.json` (или сразу пишет в БД на этапе 2).

### Этап 2. БД
3. Поднять Prisma + SQLite, описать схему из раздела 3.2.
4. Написать seed-скрипт: `sites_manifest.json` + `findings.json` → БД. Скрипт должен быть идемпотентным (можно перезапускать при добавлении новых сайтов).

### Этап 3. Next.js + shadcn
5. Инициализировать Next.js проект в текущем репозитории, подключить shadcn/ui.
6. Сделать страницу `/` — внутренний дашборд (список клиник со сводкой).
7. Сделать страницу `/admin/<clinic-id>` — детальный технический вид (заменяет текущие `sites/*.html`).

### Этап 4. Клиентский отчёт
8. Превратить `final-report-blocks.html` в React-компонент-шаблон с подстановкой данных из БД.
9. Сделать страницу `/<clinic-id>-<token>` — публичный отчёт.
10. Настроить `@media print` CSS для PDF-экспорта.

### Этап 5. Деплой
11. Залить на Vercel, привязать домен `reactortech.ru`.
12. Сгенерировать токены для всех клиник, обновить `clinics.public_token`.

### Этап 6. (Опционально) Автоматизация
13. Скрипт рассылки ссылок на `contact_email` из манифеста.
14. Серверная генерация PDF через Playwright, если ручной экспорт перестанет устраивать.

---

## 5. Что меняется в подходе

| Было | Стало |
| --- | --- |
| Сырьё → HTML напрямую | Сырьё → findings (нормализация) → HTML/React |
| Статические файлы в `sites/*.html` | Серверный рендер на актуальных данных БД |
| Шаблон отчёта правится отдельно по каждой клинике | Один шаблон на всех, изменения раскатываются автоматически |
| Чтобы обновить отчёт — запустить Python-скрипт | Поменял данные — клиент видит новое при следующем заходе |
| Один HTML на клинику (внутренний и клиентский смешаны) | Разделение: `/admin/<id>` (внутренний) и `/<id>-<token>` (клиентский) |

---

## 6. Что **не** меняется

- `scripts/audit_site.py` остаётся как есть — он собирает сырьё, и это нормально.
- Формат `data/audits/*.audit.json` не трогаем — это вход для `extract_findings.py`.
- `data/sites_manifest.json` остаётся источником правды о клиниках до миграции в БД.

---

## 7. Открытые вопросы

- Нужны ли клиенту статусы исправлений в реальном времени (галочки «исправлено»)? Это влияет на то, нужна ли админка для редактирования `findings.status` через UI.
- Нужна ли история изменений (когда уязвимость появилась, когда исправлена)? Если да — добавляем таблицу `finding_events`.
- Версионирование отчётов: показывать ли клиенту дату последнего аудита и историю прогресса?
